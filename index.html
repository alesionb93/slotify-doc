Slotify — Documentação Oficial do Produto

LEGADO Nome interno antigo: Beauty System

Slotify é um SaaS multi-tenant de agendamento e gestão operacional para salões de beleza, barbearias e estabelecimentos correlatos. Combina três experiências em uma plataforma única:

ExperiênciaPersonaCanal
Painel OperacionalEquipe interna (admin, colaborador, master_admin)Web app PWA (/agenda.html)
Booking PúblicoCliente final do salãoURL pública por tenant (/agendar/{tenantId})
Booking via WhatsAppCliente final iniciado em conversaMagic link (/agendamento-whatsapp.html?t=<token>)

1.1. Características marcantes do produto

  • Multi-tenant nativo: cada tenant possui catálogo, profissionais, clientes, tema visual, regras operacionais e usuários próprios.
  • Multi-canal de booking: o mesmo motor (agendamento-cliente.js) atende booking direto e booking via magic-link de WhatsApp.
  • Tema dinâmico por tenant: paleta gerada a partir de uma cor primária, com sincronização cross-tab via BroadcastChannel e fallback em localStorage.
  • PWA instalável com service worker custom (não usa vite-plugin-pwa).
  • Sistema de pacotes (venda + uso) integrado ao agendamento, com badges visuais e patches runtime para fluxos legados.
  • Automação de WhatsApp (Evolution API) para confirmação imediata, lembretes inteligentes (24h / 2h) e reativação de clientes inativos.
  • Push notifications Web (VAPID) para agendamentos vindos do canal externo.
  • Analytics modulares plugáveis (cancelamentos, vendas de produtos).
  • Stack vanilla JS sobre Supabase JS SDK v2 — sem framework SPA (React/Vue/Svelte).

2. Objetivo do Sistema

Centralizar e profissionalizar a operação completa de um salão, cobrindo:

  • Captação de clientes via canais públicos (link direto, WhatsApp).
  • Agendamento controlado, multi-profissional, multi-serviço, com bloqueios.
  • Atendimento com timeline visual, status (pendente/confirmado/concluído/cancelado/cancelado_com_venda) e venda de produtos no fluxo.
  • Pagamentos multi-forma (PIX, dinheiro, débito, crédito, crédito parcelado) com detecção de auto-conclusão.
  • Gestão de pessoas: profissionais, comissões, cores por profissional, vinculação serviço↔profissional.
  • Gestão de catálogo: serviços (com cor/pigmento), produtos (com custo/CMV), pacotes (venda e consumo).
  • CRM leve: clientes com histórico, importação CSV/XLSX (até 5.000 linhas), reativação automática.
  • Comunicação: WhatsApp transacional (confirmação, lembretes, reativação), push notifications.
  • Governança: roles (master_admin, admin, colaborador), feature flags por tenant, limite de usuários por tenant (3), logs de cancelamento auditáveis.
  • Analytics: KPIs de cancelamento, vendas de produtos, lucro/margem, ranking.

8.1. Roles

8.2. Resolução de role (resolveUserRoleRows)

  1. Possui master_admin → master_admin
  2. Linha com tenant_id == currentTenantId e role == 'admin' → admin
  3. Linha com tenant_id == currentTenantId e role == 'colaborador' → colaborador
  4. Fallback global/admin/colaborador → Default final: colaborador
🛡️
Limite & LoginMAX_USERS_PER_TENANT = 3. Login validado via regex /^[a-zA-Z0-9._]+$/, 3-30 chars, lowercase. Resolvido via resolve-login-to-email (anti-enumeração).

9. Fluxos de Autenticação

9.1 Fluxo de Login: /index.html → Input (login ou email) + Senha → "Entrar" ├─ Se "@" → trata como email └─ Senão → POST /functions/v1/resolve-login-to-email { login } ↓ supabaseClient.auth.signInWithPassword({ email, password }) ├─ Sucesso: Lê roles → (Múltiplos tenants? select-tenant.html : agenda.html) └─ Falha: .login-error 9.2 Fluxo de Convite/Recovery: /definir-senha.html aceita 4 formatos de token: 1. ?code=... (Magic link novo) 2. ?token_hash=... & type=invite|recovery (Clássicos) 3. #access_token=... & refresh_token=... (Hash legado) 4. ?error=... (Erro Supabase) Política: ≥8 chars, 1 maiúscula, 1 minúscula, 1 número, 1 especial. Sucesso → signOut('global') → Limpa sb-* → index.html?senha_criada=1 9.4/9.5 Criação de Tenant/Usuário: master_admin → POST /create-tenant-with-admin → Valida JWT/role → Cria tenant/auth.user/usuarios/roles → Rollback em erro admin → POST /create-user-for-tenant → Valida → countTenantUsers() (máx 3) → Cria auth.user/usuarios/roles → Rollback em erro

10. Jornadas do Usuário

10.1 Cliente final via link público

URL /agendar/{tenantId} → Valida permitir_agendamento_cliente → Step 1 (Identificação/Tel) → Step 2 (Serviço + Pacotes) → Step 3 (Profissional) → Step 4 (Calendário/Disponibilidade) → Step 5 (Confirmação + Upsell) → INSERT agendamentos → notify-whatsapp + Push → Tela de sucesso.

10.2 Cliente via WhatsApp (magic link)

Webhook → create-whatsapp-session → Resolve cliente → Gera token (30min) → Envia link via Evolution → wa_session_lookup → Pré-preenche → wa_session_mark_used → Reusa motor.

10.3/10.4 Login Interno & Pagamento

Boot → Roles/Theme/Appointments → Novo Agendamento → CRUD → INSERT (origem='interno') → Modal Pagamento (5 formas: PIX, Dinheiro, Débito, Crédito, Parcelado) → Auto-conclusão se integral → INSERT pagamentos → UPDATE status='concluido'.

11. Fluxos Completos do Sistema

11.1 Lembretes automáticos (notify-reminders v2)

▼

pg_cron a cada 5 min → POST /notify-reminders. Janela alvo: agora + 24h ± 6 min E agora + 2h ± 6 min. Inteligência: ≥24h antes → reminder_24h; 2h-24h antes → reminder_2h; <2h → skip. Anti-duplicidade via colunas reminder_*_sent_at. Atualiza DB e loga.

11.2 Reativação de inativos (notify-inactive-customers v1)

▼

pg_cron 1x/dia (10:00 BRT) → Para cada tenant com flag true: Busca clientes último agendamento > 30 dias → Filtra cooldown 30 dias → Limita 200/run → Envia via Evolution (sleep 250ms) → Loga campanhas.

11.3/11.4/11.5 Push Externo, Importação & PWA

▼

Push Externo: Trigger DB/Front → Valida origem='externo' → Busca subscriptions → webpush.sendNotification (VAPID) → 410 Gone = DELETE.
Importação: Admin → Modal → Carrega xlsx (CDN) → Parse → Preview → Normaliza Tel/Nome/Data → Dedupe → INSERT batch → Toast.
PWA Update: register(sw.js) → updateViaCache: 'none' → Throttle 60s → Toast "Atualizar" → postMessage('SKIP_WAITING') → Reload.

Origens: AVULSO (pago), PACOTE_USO (já pago), PACOTE_VENDA (paga no momento). interno vs externo (dispara WhatsApp/Push).

Feature Flags: permitir_agendamento_cliente (default false), exigir_senha_cancelamento (default true/fail-safe), inactive_customer_automation_enabled (default false).

Lembretes WhatsApp: Inteligente (24h ou 2h), Anti-duplicidade, Janela ±6 min. Reativação: 30 dias inatividade, 30 dias cooldown, 200 max/run. Tel: Normalizado BR. Usuários: Máx 3.

19. Fluxos Alternativos e de Erro

  • Login: Anti-enumeração (404 genérico), Edge offline, Body >1024b.
  • Booking: permitir_agendamento_cliente == false → Indisponível. Conflito → Constraint DB.
  • Magic Link: Token inválido/expirado/consumido → Bloqueio.
  • Cancelamento: 401 (sem JWT), 403 (role), 409 (idempotência), 401 (senha errada). Flag DB error → Fail-safe exige senha.
  • Pagamentos: Restante >0 → Concluir desabilitado. Auto-conclusão → Esconde botão.
  • PWA: Update no meio de form → Risco perda estado. Cache antigo → network-first resolve ≤3s.
